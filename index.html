Git & GitHub for Beginners

A step-by-step tutorial that takes you from "what even is version control" to comfortably pushing code, branching, and opening pull requests. Every command below is copyable — just click the Copy button.

11 Chapters Beginner Friendly Copy-Paste Ready

Ch. 1 What Are Git & GitHub?

These two terms get used interchangeably, but they're different things:

  • Git is a version control system — a program that runs on your computer and tracks changes to files over time. It lets you save "snapshots" of your project, undo mistakes, and work on multiple versions of your code at once.
  • GitHub is a website that hosts Git repositories online. It adds collaboration features on top of Git: pull requests, issues, code review, and a place to store your code in the cloud.
Analogy: Git is like the "save" and "undo history" feature for your whole project folder. GitHub is like Google Drive for that project — a place to store it online and share it with others.

Why use it?

  • Every change is saved with a message explaining what changed and why.
  • You can go back to any previous version instantly.
  • Multiple people can work on the same project without overwriting each other's work.
  • It's the industry standard — almost every software job expects you to know it.

Key vocabulary

TermMeaning
repository (repo)A project folder tracked by Git
commitA saved snapshot of your changes
branchA separate line of development
remoteA version of your repo hosted elsewhere (e.g. GitHub)
cloneDownloading a copy of a remote repo
push / pullSending / receiving changes to or from a remote

Ch. 2 Installing & Setting Up

1. Install Git

Download it from git-scm.com, or install via a package manager:

terminal — macOS (Homebrew)
brew install git
terminal — Windows (winget)
winget install --id Git.Git -e --source winget
terminal — Ubuntu/Debian
sudo apt update
sudo apt install git

2. Verify installation

terminal
git --version

3. Create a GitHub account

Sign up for free at github.com.

4. Tell Git who you are

Git stamps every commit with a name and email — set these once, globally:

terminal
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
Use the same email you signed up to GitHub with, so your commits get linked to your profile.

Ch. 3 Your First Repository

A repository ("repo") is just a folder that Git is watching for changes.

Option A — Start from scratch locally

terminal
# Create and enter a new folder
mkdir my-first-project
cd my-first-project

# Turn this folder into a Git repository
git init

This creates a hidden .git folder — that's where Git stores all its tracking data. Don't touch it directly.

Option B — Copy an existing GitHub repo

If a repo already exists on GitHub, download ("clone") it instead:

terminal
git clone https://github.com/username/repository-name.git

Check the status any time

This is the command you'll run constantly — it tells you what's changed:

terminal
git status

Ch. 4 The Core Workflow

Almost everything you do in Git follows the same three-step loop:

  1. Edit files in your project
  2. Stage the changes you want to save (git add)
  3. Commit them as a snapshot (git commit)

1. Make a change

terminal
echo "# My First Project" > README.md

2. Stage the file

Staging tells Git "include this file in my next snapshot."

terminal — stage a single file
git add README.md
terminal — stage everything changed
git add .

3. Commit the snapshot

A commit is permanent (well, recoverable) — always write a clear message describing what changed.

terminal
git commit -m "Add initial README"

4. View history

terminal — full log
git log
terminal — condensed one-line log
git log --oneline
Habit to build: git status → git add → git commit -m "...". Repeat this every time you finish a small chunk of work.

Ch. 5 Connecting to GitHub

So far everything has been local — only on your computer. Now let's push it to GitHub so it's backed up online and shareable.

1. Create an empty repo on GitHub

On github.com/new, create a new repository. Don't initialize it with a README if you already have a local project — that avoids conflicting histories.

2. Link your local repo to it

terminal
git remote add origin https://github.com/username/repository-name.git

origin is just a nickname for that remote URL — you'll use it in future commands.

3. Push your commits

terminal — first push
git branch -M main
git push -u origin main

The -u flag links your local main branch to GitHub's, so future pushes can just be git push.

Everyday sync commands

terminal — send local commits to GitHub
git push
terminal — fetch and merge changes from GitHub
git pull
Authentication: GitHub no longer accepts passwords over HTTPS. Use a Personal Access Token, GitHub CLI (gh auth login), or SSH keys instead.

Ch. 6 Branching & Merging

A branch is an independent line of work. The default branch is usually called main. Create a new branch whenever you start a feature or fix, so main always stays stable.

Create and switch to a new branch

terminal
git checkout -b feature/login-page

Modern Git also supports the newer switch command:

terminal
git switch -c feature/login-page

See all branches

terminal
git branch

Switch between branches

terminal
git checkout main

Merge a branch into main

Once your feature is done, switch to main and merge it in:

terminal
git checkout main
git merge feature/login-page

Delete a branch you no longer need

terminal
git branch -d feature/login-page
Merge conflicts: if two branches changed the same lines, Git will pause the merge and mark the conflicting sections with <<<<<<< / ======= / >>>>>>>. Edit the file to keep the correct version, then run git add and git commit to finish the merge.

Ch. 7 Collaborating with Others

Forking

A fork is your own copy of someone else's repository, made under your GitHub account. Use it when you don't have write access to the original project — click "Fork" on GitHub, then clone your copy:

terminal
git clone https://github.com/your-username/their-repo.git

Pull Requests (PRs)

A pull request asks the original project's maintainers to merge your changes. The typical flow:

  1. Create a branch for your change
  2. Commit and push it to your fork/repo
  3. Open a PR on GitHub comparing your branch to their main
  4. Maintainers review, comment, and merge (or request changes)
terminal — push a feature branch so you can open a PR
git push -u origin feature/login-page

Then click "Compare & pull request" on GitHub.

Keeping your fork up to date

terminal
git remote add upstream https://github.com/original-owner/original-repo.git
git fetch upstream
git merge upstream/main

Issues

Issues are GitHub's built-in tracker for bugs, feature requests, and discussion. Anyone with access to a repo can open one from the "Issues" tab — no Git commands needed.

Ch. 8 .gitignore & Best Practices

Some files should never be committed: secrets, build artifacts, dependency folders, OS clutter. A .gitignore file tells Git to skip them.

Create one

terminal
touch .gitignore

Common example (Node.js project)

.gitignore
node_modules/
.env
.DS_Store
dist/
*.log

Good habits

  • Write clear, specific commit messages: "Fix null pointer in cart total" beats "fix stuff".
  • Commit often, in small logical chunks — easier to review and to undo.
  • Never commit secrets (API keys, passwords). Use .gitignore and environment variables instead.
  • Pull before you push, to avoid conflicts.
  • Use branches for anything beyond a trivial fix.

Ch. 9 Fixing Mistakes

Undo changes in a file (not yet staged)

terminal
git checkout -- filename.txt

Unstage a file (keep the edits)

terminal
git restore --staged filename.txt

Edit your last commit message

terminal
git commit --amend -m "Corrected commit message"

Undo the last commit but keep the changes

terminal
git reset --soft HEAD~1

Discard the last commit completely

terminal — destructive, use with care
git reset --hard HEAD~1
Warning: --hard permanently deletes uncommitted changes. Double-check git status before using it.

See what changed before committing

terminal
git diff

Ch. 10 Advanced: Merge & Branch Troubleshooting

Once you're past the basics, branching and merging start throwing errors that feel scary the first time you see them. None of them are dangerous if you know what's actually happening. This chapter walks through the situations that trip up almost every beginner.

How a merge actually looks

When you merge a feature branch into main, Git creates a special merge commit that has two parents — it stitches the two histories back together.

Merge vs. rebase

These are the two ways to bring a feature branch's changes back into main. Both end with the same code — they just record history differently.

terminal — rebase your feature branch onto the latest main
git checkout feature/login-page
git fetch origin
git rebase origin/main

Resolving a merge conflict, step by step

A conflict happens when two branches edit the same lines of the same file. Git stops and asks you to decide. Opening the file, you'll see markers like this:

example.js — conflicted file
<<<<<<< HEAD
const greeting = "Hello, user!";
=======
const greeting = "Welcome back!";
>>>>>>> feature/login-page
  • Everything between <<<<<<< HEAD and ======= is your current branch's version.
  • Everything between ======= and >>>>>>> branch-name is the incoming version.

Edit the file down to the version you actually want (deleting all three marker lines), then finish the merge:

terminal
# See which files are conflicted
git status

# After manually fixing the file(s):
git add example.js
git commit -m "Resolve merge conflict in example.js"
Common mistake: forgetting to remove the <<<<<<< / ======= / >>>>>>> markers themselves before committing. Always search the file for <<<<<<< before you stage it.

If a merge goes badly, back out entirely and try again:

terminal — abort an in-progress merge
git merge --abort

Conflicts during a rebase

Rebase conflicts work the same way, but Git pauses on each replayed commit instead of once:

terminal
# Fix the conflicted file(s), then:
git add .
git rebase --continue

# Or bail out completely and return to before the rebase:
git rebase --abort

"Your branch and 'origin/main' have diverged"

This shows up when you and someone else both pushed commits to the same branch. Git refuses to push, since it can't tell which history should win:

terminal output
! [rejected]        main -> main (fetch first)
error: failed to push some refs
hint: Updates were rejected because the remote contains work that you do not have locally.

Fix it by pulling first — either with a merge commit or, for a cleaner line, with rebase:

terminal
git pull --rebase
git push
Never reach for git push --force to make this error go away — it overwrites the remote history, including your teammate's commits. If you must force-push (e.g. after your own rebase), use the safer flag below, which fails instead of overwriting if someone else pushed in the meantime:
terminal
git push --force-with-lease

Detached HEAD state

HEAD is just a pointer to "where you currently are." Normally it points at a branch name, which points at a commit. If you check out a specific commit hash (or an old tag) instead of a branch, HEAD points straight at that commit — you're in a "detached" state.

Common mistake: checking out an old commit to look at it, making edits and committing, then switching branches — those commits aren't on any branch and become hard to find. Before switching away, save them:
terminal — turn a detached HEAD into a real branch
git switch -c rescued-work

Stashing work in progress

Need to switch branches but aren't ready to commit? git stash shelves your uncommitted changes so your working directory is clean, then you can bring them back later.

terminal
# Shelve current changes
git stash

# ... switch branches, do other work ...

# Bring the changes back
git stash pop
terminal — see all stashed changes
git stash list

git reflog — your safety net

Git almost never truly deletes a commit right away, even after a hard reset, a bad rebase, or a deleted branch. reflog keeps a local log of everywhere HEAD has pointed, so you can recover from nearly any mistake.

terminal
# List recent HEAD movements with their commit hashes
git reflog

# Recover by creating a branch at a specific point
git checkout -b recovered-branch <commit-hash>

Common mistakes & how to fix them

MistakeSymptomFix
Committed straight to main Unreviewed changes on the shared branch git branch save-work to capture it, then reset main to match the remote
Push rejected — branch diverged ! [rejected] ... (fetch first) git pull --rebase then git push
Merged the wrong branch Unexpected files/changes appear after merge git reset --hard ORIG_HEAD right after the merge
Force-pushed over a teammate's work Their commits vanish from the remote Use --force-with-lease going forward; recover lost commits from their reflog
Left conflict markers in a commit Code won't compile; <<<<<<< visible in file Search the repo for <<<<<<<, fix, commit again
Committed a secret or API key Key exposed in commit history Rotate the secret immediately; scrub history with git filter-repo or the BFG Repo-Cleaner
Deleted a branch too early "branch not found" but you need those commits git reflog to find the last commit hash, then git checkout -b from it
Committed while in detached HEAD Commits disappear after switching branches git switch -c new-branch before switching away
